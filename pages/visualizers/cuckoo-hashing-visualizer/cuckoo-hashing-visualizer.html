Two tables. Two hashes. Never more than 2 checks to find anything.

Every other hash table gives you average O(1) lookup with no worst-case guarantee. Cuckoo hashing gives you a real one: every key lives in exactly one of two possible slots. Inserting can kick out the resident key like a cuckoo chick pushing eggs from a nest — that evicted key then tries its own second table, potentially chaining several evictions deep.

🐦 How the Nest Works

🎯

Exactly 2 Possible Homes

Every key has exactly one valid slot in Table 1 (via h1) and one valid slot in Table 2 (via h2). Lookup checks both — never more, regardless of how full or contended the tables are. That's the genuine O(1) worst case.

🔄

The Eviction Chain

If a key's target slot is occupied, the resident is evicted and the new key takes its place. The evicted key then tries ITS other table via its own second hash — this can chain several evictions deep before everything settles.

♻️

Cycle Detection & Rehash

Occasionally an eviction chain loops back on itself forever. After a bounded number of kicks, the algorithm gives up, generates two brand-new hash functions, and re-inserts every key from scratch — a full rehash.

📊

The 50% Load Factor Wall

Cuckoo hashing works cleanly up to roughly 50% load factor. Beyond that, eviction chains grow long and rehashes become frequent — a real tradeoff against chaining-based hash tables, which degrade more gracefully but never offer the O(1) worst-case guarantee.
