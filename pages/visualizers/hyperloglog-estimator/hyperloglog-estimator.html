HyperLogLog turns this single strange fact into a ~2% accurate cardinality estimator using a fixed, tiny register array.

📊 Why Leading Zeros Encode Cardinality

🪙

The Coin Flip Intuition

A random hash bit is like a coin flip. The probability of seeing k leading zeros in a row is 1/2^k. If you've hashed N items and the longest run seen is k zeros, then N is probably around 2^k — because it took about that many "flips" to get that lucky.

📦

Why m Registers Instead of 1

A single register has huge variance — one lucky item can throw off the whole estimate. Splitting into m buckets by hash prefix and taking the harmonic mean of 2^register across all of them averages out the luck, bringing error down to about 1.04/√m.

➗

Why Harmonic Mean, Not Arithmetic

A single register with an unusually large run length would dominate and badly skew an arithmetic mean. The harmonic mean of 2^register values is far more robust to that one outlier bucket, which is exactly why Flajolet et al. chose it in the original 2007 paper.

💾

Real Deployments

Redis's PFCOUNT uses HyperLogLog with 16384 registers (~12KB) to estimate cardinality of sets with billions of elements. Google BigQuery's APPROX_COUNT_DISTINCT and Presto's approx_distinct use the same core algorithm for exactly this reason: constant memory regardless of stream size.
