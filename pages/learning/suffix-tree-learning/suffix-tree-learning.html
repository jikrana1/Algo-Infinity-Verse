Suffix Trees

From intuition to Ukkonen — build the idea, then the data structure.

Motivation Terminology Naive Ukkonen Practice Quiz

Learning Path

Mental model

A suffix tree is a compressed trie of all suffixes, built to answer substring queries fast. Ukkonen’s algorithm builds it incrementally, reusing work via suffix links.

Motivation

Suppose you want to answer many questions of the form: “Does pattern P occur in text T?” or “Where does it occur?”. Suffix trees turn these substring problems into traversals of a tree built from T.

Substring queries reduce to walking edges.
Longest common prefixes come “for free” from paths.
Many applications: repeated substrings, LCP, palindrome tricks.

Terminology

Suffix

For text T of length n, the suffix starting at index i is T[i..n-1].

Suffix trie

A trie containing all suffixes. It’s huge, but conceptually simple.

Edge labels

In a suffix tree, edges store intervals into T, not explicit strings. This is why it’s space-efficient.

Suffix link

A pointer that lets Ukkonen jump between “almost the same” partial matches. It connects internal nodes representing different active states.

Active point

Ukkonen’s algorithm tracks where it is in the current construction. (Think: “the state of the partial suffix being added.”)

Implicit vs explicit

You may be located within an edge (implicit). Nodes are explicit endpoints (explicit).

The naive path is: suffix trie → compress edges → obtain a suffix tree.

Naive idea (high-level pseudo-code) 
buildSuffixTreeNaive(T):
  trie = emptyTrie()
  for i in [0..n-1]:
    insertSuffix(trie, T[i..])

  // Compress chains of degree-1 nodes into single edges
  tree = compressTrie(trie)
  return tree

Ukkonen's Algorithm

Ukkonen builds the suffix tree in O(n) time by adding one character at a time while maintaining an active point and using suffix links to avoid recomputation.

Big picture loop

  • For each new prefix T[0..i], extend all suffixes that need it.
  • Use the active point to know where to attempt extension.
  • If extension already exists, stop early (rule 3 in many descriptions).

Role of suffix links

Suffix links connect internal nodes that represent similar partial substrings. When you fail to extend from one state, you jump to another state instead of restarting.

Why it’s linear

Each time you create an edge split or a new leaf, the algorithm progresses. The number of “expensive” operations is bounded, so total work is O(n).

Ukkonen (rule-based summary) 
Repeat for each extension i:
  while extensionsToAdd > 0:
    try extend from active point
    if next character matches:
      (Rule 3) increment showstopper and break
    else if edge ends / need split:
      create split node + new leaf (Rule 2)
      follow suffix link
    else if node exists but no edge:
      create new leaf (Rule 1)

  move to next i
  update active point

Applications

Substring search

Traverse edges by the pattern; all occurrences relate to the subtree.

Repeated substrings

Deep paths correspond to common prefixes among suffixes.

LCP / LCS

Compute longest common prefix between suffixes via nearest common ancestor.

Palindrome tricks

With clever transformations (e.g., separators), suffix structures help find repeats.

Complexity

Metric Suffix Tree
Construction time O(n) using Ukkonen
Construction space O(n) edges store intervals
Pattern match O(m) (m = pattern length)

Practice Problems

Practice is the fastest way to internalize suffix trees. Start with tasks that check substring logic, then move to problems involving LCP/LCS and repeated substrings.

Quick drills

  • Given T and P, reason about how many suffixes start with P.
  • Compute LCP(T[i..], T[j..]) by hand for small strings.
  • Find the longest repeated substring in a short string.

“Tree thinking” prompts

  • Which internal nodes correspond to prefixes that repeat?
  • How would suffix links help when you fail to extend?
  • When does Ukkonen stop early?
